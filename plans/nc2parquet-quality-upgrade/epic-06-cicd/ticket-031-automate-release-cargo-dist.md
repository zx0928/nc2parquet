# ticket-031 Automate Release Process with cargo-dist

## Context

### Background

The nc2parquet project already has a functional cargo-dist release pipeline. The auto-generated `release.yml` workflow (from cargo-dist v0.30.0) handles plan/build/host/announce on version tag push, and `publish.yml` handles crates.io publishing. However, the current setup has gaps: it only targets two Linux architectures (x86_64 and aarch64), has no macOS targets, does not generate release notes from the CHANGELOG, and the `Cargo.toml` metadata needs verification for crates.io readiness. This ticket audits and enhances the existing release automation to cover all target platforms and integrate with the CHANGELOG created in Epic 05.

### Relation to Epic

This is the third ticket in Epic 06. It depends on ticket-032 (cross-compilation CI matrix) for target platform validation, and builds on the CHANGELOG created in ticket-025. The release pipeline is the final piece that turns a git tag into downloadable binaries, a GitHub Release with structured notes, and a crates.io publication.

## Specification

### Requirements

1. **Update `dist-workspace.toml` targets** to add macOS targets: `x86_64-apple-darwin` and `aarch64-apple-darwin`. This gives four targets total: linux-x86_64, linux-aarch64, macos-x86_64, macos-aarch64.
2. **Regenerate `release.yml`** by running `cargo dist init` to pick up the new targets. The regenerated workflow will include macOS runner entries in the build matrix. After regeneration, verify the workflow structure is correct and no unwanted changes were introduced.
3. **Configure release notes** in `dist-workspace.toml`: set `changelog = true` so cargo-dist extracts the current version's section from CHANGELOG.md for the GitHub Release body.
4. **Verify crates.io readiness**: Confirm all required `Cargo.toml` metadata fields are present (already verified in Current State above -- no changes needed).
5. **Do NOT add Windows targets**: The netcdf crate's static build feature is poorly tested on Windows (documented in CONTRIBUTING.md). Windows can be added in a future iteration when community demand exists.
6. **Do NOT change the publishing workflow** (`publish.yml`): It already works correctly with `cargo-release` and `CRATES_IO_API_TOKEN`. Leave it as-is.
7. **Update CHANGELOG.md**: When this ticket is implemented, the `[Unreleased]` section should remain as-is (the actual version bump and CHANGELOG split happens at release time, not at CI configuration time).

### Inputs/Props

- **cargo-dist version**: 0.30.0 (already pinned in `dist-workspace.toml`)
- **Tag format**: `v0.2.0` style (the existing release.yml already matches `**[0-9]+.[0-9]+.[0-9]+*` which covers both `v0.2.0` and `0.2.0`)
- **GitHub secrets required**: `GITHUB_TOKEN` (built-in, used by release.yml), `CRATES_IO_API_TOKEN` (used by publish.yml, already configured)

### Outputs/Behavior

- On tag push matching a version pattern: cargo-dist builds binaries for all four targets, creates a GitHub Release with release notes extracted from CHANGELOG.md, uploads tarballs with shell installer scripts.
- On PR: cargo-dist runs a dry-run `plan` step only (no artifacts built), validating that the dist configuration is correct.

### Error Handling

- If cargo-dist cannot build for a target (e.g., macOS cross-compilation fails), the specific matrix entry fails and the release is blocked. This is correct -- partial releases should not be created.
- If CHANGELOG.md is missing or has no section for the tagged version, cargo-dist falls back to auto-generated release notes from git history.

## Implementation Guide

### Suggested Approach

1. **Update `/home/rogerio/git/nc2parquet/dist-workspace.toml`**:
   - Add `x86_64-apple-darwin` and `aarch64-apple-darwin` to the `targets` array.
   - Add `changelog = true` to the `[dist]` section.
   - The updated file should look like:

   ```toml
   [workspace]
   members = ["cargo:."]

   [dist]
   cargo-dist-version = "0.30.0"
   ci = "github"
   installers = ["shell"]
   targets = [
       "aarch64-unknown-linux-gnu",
       "x86_64-unknown-linux-gnu",
       "x86_64-apple-darwin",
       "aarch64-apple-darwin",
   ]
   install-path = "CARGO_HOME"
   install-updater = false
   changelog = true
   ```

2. **Regenerate the release workflow**: Run `cargo dist init --yes` to regenerate `.github/workflows/release.yml` with the updated target list. Review the diff to ensure only the expected changes were made (new macOS build matrix entries).

3. **Verify locally**: Run `cargo dist plan` and confirm it lists all four targets in the output manifest.

4. **Update CONTRIBUTING.md**: In the CI pipeline table (around line 420), ensure the existing description of the release process mentions that cargo-dist builds for Linux (x86_64, aarch64) and macOS (x86_64, aarch64).

### Patterns to Follow

- The `release.yml` file is auto-generated by cargo-dist. Do NOT hand-edit it beyond what `cargo dist init` produces. If customization is needed, use `dist-workspace.toml` configuration options.
- Keep the `publish.yml` completely untouched. It is a separate concern (crates.io) from the binary release (GitHub Releases).

### Pitfalls to Avoid

- **Do NOT hand-edit `release.yml`**: It is auto-generated and will be overwritten by future `cargo dist init` runs. All configuration goes in `dist-workspace.toml`.
- **Do NOT add `installers = ["homebrew"]` yet**: Homebrew tap publishing requires a separate tap repository and additional configuration. Start with the shell installer only.
- **Do NOT add Windows targets**: The netcdf crate's static build on Windows is unreliable. This is documented in CONTRIBUTING.md and should be a separate future initiative.
- **Do NOT modify `publish.yml`**: It already works. Touching it risks breaking crates.io publishing.
- **Verify `cargo dist init` does not downgrade the cargo-dist version**: The `cargo-dist-version` in `dist-workspace.toml` should remain at `0.30.0` unless a newer version is explicitly desired.
- **Run `cargo dist plan` locally before committing**: This catches configuration errors before they hit CI.
