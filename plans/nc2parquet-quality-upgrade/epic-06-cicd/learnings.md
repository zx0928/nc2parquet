# Epic 06 Learnings: CI/CD and Release Quality

## Patterns Established

- **Separate benchmark workflow with path filters**: The benchmark regression pipeline lives in a dedicated `.github/workflows/benchmark.yml` rather than in `ci.yml`. This keeps the main CI fast by avoiding 2-5 minute benchmark runs on every push. Path filters (`src/**`, `benches/**`, `Cargo.toml`, `Cargo.lock`) ensure the workflow only runs when source or benchmark code changes, skipping docs-only PRs entirely. See `/home/rogerio/git/nc2parquet/.github/workflows/benchmark.yml`.

- **`include`-based CI matrix for selective platform/toolchain pairs**: The `test` job uses `matrix.include` entries rather than a Cartesian product of `os` x `rust`. This gives precise control: Ubuntu runs both stable and beta, macOS runs stable only. A Cartesian product would create an unwanted macOS/beta combination that adds cost without proportional benefit. Combined with `fail-fast: false`, a failure on one platform does not cancel the others. See `/home/rogerio/git/nc2parquet/.github/workflows/ci.yml` lines 17-25.

- **Conditional system dependency installation by OS**: Platform-specific `apt-get` and `brew` steps are gated with `if: runner.os == 'Linux'` and `if: runner.os == 'macOS'`. This is cleaner than separate workflow files and keeps all test logic in one place. The macOS step uses only `brew install netcdf` — Homebrew installs both the library and headers with a single formula. See `ci.yml` lines 47-55.

- **OS-namespaced cache keys prevent cross-platform cache poisoning**: The cache key pattern `${{ runner.os }}-cargo-${{ hashFiles('**/Cargo.lock') }}` guarantees that Ubuntu and macOS builds never share a `target/` directory. Without the `runner.os` prefix, platform-specific compiled artifacts would cause silent link-time failures on the wrong OS. The coverage job uses a separate `-cargo-cov-` suffix to prevent its `--skip-clean` behavior from polluting the test job's incremental cache. See `ci.yml` lines 45 and 90.

- **`taiki-e/install-action` for pre-built tool binaries**: The coverage job installs `cargo-tarpaulin` via `uses: taiki-e/install-action@cargo-tarpaulin` rather than `cargo install cargo-tarpaulin`. The `install-action` downloads a pre-built binary, reducing the coverage job's setup time by 2-3 minutes compared to compiling from source. This pattern applies to any frequently-installed Cargo tool that has pre-built releases. See `ci.yml` line 98.

- **`--skip-clean` on tarpaulin preserves incremental compilation**: Running `cargo tarpaulin --skip-clean` avoids a full rebuild on every coverage run. Without this flag, tarpaulin forces a clean build each time (because instrumentation changes object files), making the job unnecessarily slow. The separate cache key suffix (`-cargo-cov-`) ensures that tarpaulin's instrumented artifacts do not pollute the non-instrumented test job cache. See `ci.yml` line 102.

- **Soft benchmark alerting (warning comments, not CI failures)**: The benchmark workflow sets `fail-on-alert: false` and `comment-on-alert: true` with an `alert-threshold` of `"150%"`. GitHub Actions runner performance varies by 5-10% between runs, making a hard failure threshold produce flaky CI. A 150% threshold (1.5x slower) is wide enough to tolerate noise while still catching genuine regressions. Alerts appear as PR comments, not red checks. See `/home/rogerio/git/nc2parquet/.github/workflows/benchmark.yml` lines 57-60.

- **`auto-push` conditional on event type for benchmark baselines**: The `github-action-benchmark` action is configured with `auto-push: ${{ github.event_name == 'push' }}`. This means baseline data is written to the `gh-pages` branch only on pushes to main, not on PRs. PRs read the baseline for comparison but do not update it, preventing feature branches from corrupting the main performance history. See `benchmark.yml` line 56.

- **cargo-dist `dist-workspace.toml` as the single source of truth for release targets**: All target triples and release configuration live in `dist-workspace.toml`. The generated `release.yml` is explicitly auto-generated and must not be hand-edited. When targets are added, the workflow is regenerated by running `cargo dist init`. This separation prevents the two files from drifting out of sync. See `/home/rogerio/git/nc2parquet/dist-workspace.toml` and `/home/rogerio/git/nc2parquet/.github/workflows/release.yml` line 1 (autogeneration notice).

- **Explicit `cargo test --doc` step in CI**: Doc-tests are now run as an explicit named step ("Run doc-tests") separate from "Run library tests". This gives direct failure attribution in the GitHub Actions UI: if a public API example stops compiling, the failure shows on the doc-test step, not buried in a monolithic test run. The step runs on all matrix entries (Ubuntu stable, Ubuntu beta, macOS stable). See `ci.yml` lines 66-67.

- **Codecov `fail_ci_if_error: false` as mandatory setting**: The Codecov upload action sets `fail_ci_if_error: false` to prevent Codecov service outages from blocking PRs. Coverage measurement failures (tarpaulin crashes, test failures) still fail the job through the normal `cargo tarpaulin` exit code — only the upload step is made non-fatal. See `ci.yml` line 110.

## Architectural Decisions

- **cargo-tarpaulin over cargo-llvm-cov for coverage measurement**: The ticket specification explicitly chose `cargo-tarpaulin` because it uses ptrace-based instrumentation on pre-built binaries, avoiding a full rebuild with LLVM instrumentation flags. `cargo-llvm-cov` requires rebuilding all dependencies with `-C instrument-coverage`, which can fail with the netcdf crate's C library FFI due to incompatible compiler flags during the static linking phase. This decision is baked into `ci.yml` and should not be reversed without validating that `cargo-llvm-cov` works with `netcdf = { features = ["static"] }` on the target runner.

- **Coverage exclusions include handlers, test helpers, and test modules**: The tarpaulin command excludes `src/main.rs`, `src/handlers/*`, `src/test_helpers.rs`, `src/tests/*`, and `benches/*`. Excluding handlers and test infrastructure prevents coverage inflation from CLI glue code and test scaffolding, which are not meaningful signal for library correctness. The coverage percentage reported by Codecov reflects only the core library modules (`src/extract.rs`, `src/filters.rs`, `src/postprocess.rs`, `src/output.rs`, `src/input.rs`, `src/errors.rs`, `src/storage.rs`). See `ci.yml` line 103.

- **macOS added to CI matrix but not Windows**: Windows was explicitly excluded because the `netcdf` crate's static build feature is poorly tested on Windows (requires manual `HDF5_DIR` and `NETCDF_DIR` environment variable setup, documented in `CONTRIBUTING.md` lines 95-101). macOS uses `macos-latest` (Apple Silicon arm64 runner) rather than a pinned `macos-13` to avoid version brittleness. The aarch64-unknown-linux-gnu target is validated at release time via cargo-dist's container-based cross-compilation, not in the CI matrix.

- **Four release targets: linux x86_64/aarch64 + macOS x86_64/aarch64**: The `dist-workspace.toml` targets array covers all four production deployment targets. Windows was excluded (same rationale as CI matrix). A Homebrew installer was not added because it requires a separate tap repository configuration. The `changelog = true` setting connects the GitHub Release body to `CHANGELOG.md`, eliminating auto-generated release notes from git history. See `/home/rogerio/git/nc2parquet/dist-workspace.toml`.

- **Integration tests stay in `cargo test --lib`, not a separate CI job**: The ticket-033 specification deliberately chose not to create a separate integration test job or a `tests/` directory. The `src/tests/test_integration.rs` tests run as `#[cfg(test)]` library tests because they need `pub(crate)` access to internal types. A `tests/` directory is for external integration tests that import the library as a black box. The distinction matters: if tests need access to crate internals, they belong in `src/tests/`, not `tests/`.

## Conventions Adopted

- **Never hand-edit `release.yml`**: The autogeneration notice on line 1 of `release.yml` establishes this contract. All release configuration changes go into `dist-workspace.toml` only; `cargo dist init` regenerates the workflow. Editing `release.yml` directly creates a file that will be silently overwritten on the next `cargo dist init` run.

- **Separate cache key suffix per job type**: The test job uses `-cargo-${{ hashFiles('**/Cargo.lock') }}`, the coverage job uses `-cargo-cov-${{ hashFiles('**/Cargo.lock') }}`, and the benchmark job uses `-bench-${{ hashFiles('**/Cargo.lock') }}`. This three-way separation prevents instrumented (tarpaulin) and profiled (bench) artifacts from entering the standard incremental compilation cache.

- **Benchmark flags position: after `--` separator**: The Criterion `--output-format=bencher` flag must be placed after the `--` separator in `cargo bench` invocations, because it is a flag for the benchmark binary itself, not for `cargo`. The command in `benchmark.yml` line 47 is: `cargo bench --bench name1 --bench name2 -- --output-format=bencher`. Future benchmarks added via `--bench nameN` go before the `--`; any Criterion-level flags go after.

- **tarpaulin `--lib` flag prevents `dhat-heap` feature activation**: Running `cargo tarpaulin --lib` (without `--all-features`) avoids loading the `dhat-heap` feature's global allocator, which conflicts with tarpaulin's ptrace instrumentation. Adding `--all-features` would break the coverage job silently if the `dhat-heap` feature is enabled. The `--lib` flag is the correct scope for coverage of the library crate.

## Surprises and Deviations

- **Ticket-033 title changed from "integration test CI job" to "doc-test CI step"**: The epic overview described this ticket as "Add a dedicated integration test CI job with test NetCDF fixtures." The actual implementation determined that the integration tests in `src/tests/test_integration.rs` already run under `cargo test --lib` (they are `#[cfg(test)]` library-internal tests that require `pub(crate)` access). No separate job was warranted. The ticket was refocused on adding the missing doc-test step and renaming the existing "Run unit tests" step to "Run library tests" to accurately reflect that `cargo test --lib` runs all library tests, including integration tests.

- **Codecov tokenless upload acceptable for public repos, token added for reliability**: The ticket specification noted that Codecov supports tokenless upload for public repositories but recommended adding `CODECOV_TOKEN` for reliability. The implementation includes `token: ${{ secrets.CODECOV_TOKEN }}` in the upload step. If the secret is not configured, the upload falls back to tokenless mode — Codecov degrades gracefully. The `fail_ci_if_error: false` guards against both the missing-token and network-failure scenarios.

- **benchmark.yml uses `tool: cargo` not `tool: criterion`**: The `github-action-benchmark` action's `tool` parameter for Criterion benchmarks using `--output-format=bencher` is set to `"cargo"`, not `"criterion"`. The bencher output format is a Cargo-ecosystem JSON standard that the action understands when tool is `"cargo"`. Using `"criterion"` would require HTML report parsing, which is a different output mode.

- **CONTRIBUTING.md CI table gained a Platform column**: The original ticket specification showed a two-column table (Job, Command). The implementation added a third "Platform" column listing "Ubuntu + macOS" or "Ubuntu" for each job. This was necessary because the CI matrix expansion meant the same jobs run on different platform subsets, and documenting that distinction helps contributors understand why a macOS-only failure does not also show an Ubuntu failure.

## Recommendations for Future Work

- **Add `cargo doc --no-deps` check to CI**: Currently the CI does not verify that `cargo doc` builds without warnings. Adding a `cargo doc --no-deps -- -D warnings` step to the `test` job would catch broken doc-comment links and missing cross-references before they reach crates.io. The coverage job's exclusions already provide a model for how to scope the doc check to library code only.

- **Consider pinning `actions/cache@v3` to `@v4`**: The three workflow files (`ci.yml`, `benchmark.yml`) all use `actions/cache@v3`. GitHub is actively maintaining v4 with improved cache performance. Updating all three simultaneously (changing the version string in `ci.yml` lines 37 and 81, and `benchmark.yml` line 33) keeps the workflows on a supported version.

- **Benchmark baseline initialization is manual**: The `github-action-benchmark` action will auto-create the `gh-pages` branch and initial baseline on the first push to main. However, this first run has no baseline to compare against, so no regression alerts fire. To prime the baseline before CI regressions are meaningful, merge the `benchmark.yml` workflow file and push a commit to main — the first run populates `dev/bench/` in `gh-pages`. Subsequent PRs then have a real baseline.

- **`CRATES_IO_API_TOKEN` secret must be configured for `publish.yml`**: The `publish.yml` workflow (untouched by this epic) requires a `CRATES_IO_API_TOKEN` GitHub Actions secret. Before the first `v0.2.0` release, this secret must be populated in the repository settings. The secret is not required for CI checks or benchmark runs; it is only consumed by the `cargo-release` publish step triggered on version tags.

- **macOS Homebrew cache is not cached across runs**: The `brew install netcdf` step in the macOS CI matrix entry runs unconditionally on every job (Homebrew is not included in the `actions/cache` path list). Adding `~/.cache/Homebrew` to the cache paths (with a key based on `runner.os` and a fixed Homebrew lock-file hash) could reduce the macOS job setup time by 30-60 seconds per run. This is a cost-optimization, not a correctness issue.
