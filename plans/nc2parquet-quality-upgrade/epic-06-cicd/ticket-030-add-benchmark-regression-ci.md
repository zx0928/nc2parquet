# ticket-030 Add Benchmark Regression Detection in CI

## Context

### Background

Epic 03 established four Criterion benchmark suites (`extraction_bench`, `filter_bench`, `postprocess_bench`, `combination_bench`) covering the hot paths in the extraction pipeline, filter application, post-processing, and dimension combination iteration. These benchmarks run locally via `cargo bench`, but there is no CI automation to detect performance regressions on pull requests. A developer can accidentally introduce a 30% slowdown in the extraction pipeline and the CI will pass green. This ticket adds automated benchmark regression detection.

### Relation to Epic

This is the second ticket in Epic 06. It complements the coverage reporting (ticket-029) by adding performance as a second quality signal in CI. Together, coverage and benchmark regression detection form the two quality gates that protect code correctness and performance.

## Specification

### Requirements

1. Create a new workflow file `.github/workflows/benchmark.yml` (separate from `ci.yml` to avoid slowing down the main CI pipeline).
2. The workflow triggers on pull requests targeting `main` only when source files change (not docs-only PRs). Use a `paths` filter: `src/**`, `benches/**`, `Cargo.toml`, `Cargo.lock`.
3. Use the `github-action-benchmark` action (`benchmark-action/github-action-benchmark@v1`) to compare benchmark results against a stored baseline.
4. Store benchmark baselines in the `gh-pages` branch under a `dev/bench` directory. This is the standard approach used by `github-action-benchmark` and avoids cache eviction issues.
5. On PRs: run benchmarks, compare against the `gh-pages` baseline, and post a comment if any benchmark regresses by more than 15%. Do NOT fail the CI check -- post a warning comment only.
6. On pushes to `main`: run benchmarks and update the `gh-pages` baseline (this keeps the baseline fresh with each merge).
7. Use Criterion's JSON output format (`--output-format=bencher`) for compatibility with `github-action-benchmark`.
8. Run benchmarks on `ubuntu-latest` with `stable` Rust only, pinned to a specific runner to reduce noise. Use the `runs-on: ubuntu-latest` runner (GitHub's standard 2-core runner provides adequate consistency for regression detection at a 15% threshold).

### Inputs/Props

- **GitHub token**: `${{ secrets.GITHUB_TOKEN }}` with write permission to the `gh-pages` branch for baseline updates.
- **Alert threshold**: 150% (i.e., a benchmark result that is 1.5x slower than baseline triggers a warning comment).
- **Benchmark output**: Criterion's `--output-format=bencher` produces JSON lines compatible with `github-action-benchmark`.

### Outputs/Behavior

- On pull requests: a GitHub comment is posted listing any benchmarks that regressed beyond the threshold, with before/after timings.
- On main pushes: the `gh-pages` branch is updated with the latest benchmark data under `dev/bench/`.
- A benchmark trend chart is available at `https://rjmalves.github.io/nc2parquet/dev/bench/` (auto-generated by `github-action-benchmark`).

### Error Handling

- If `cargo bench` fails (compilation error, test fixture missing), the workflow job fails, making the benchmark check red on the PR.
- If the `gh-pages` branch does not exist yet (first run), `github-action-benchmark` creates it automatically.
- If no baseline exists yet (first run on a new repo), the comparison step is skipped and the results are stored as the initial baseline.

## Implementation Guide

### Suggested Approach

1. **Create `/home/rogerio/git/nc2parquet/.github/workflows/benchmark.yml`**:

```yaml
name: Benchmark

on:
  push:
    branches: [main]
    paths: ["src/**", "benches/**", "Cargo.toml", "Cargo.lock"]
  pull_request:
    branches: [main]
    paths: ["src/**", "benches/**", "Cargo.toml", "Cargo.lock"]

permissions:
  contents: write
  pull-requests: write

env:
  CARGO_TERM_COLOR: always

jobs:
  benchmark:
    name: Performance Regression Check
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install Rust
        uses: dtolnay/rust-toolchain@v1
        with:
          toolchain: stable

      - name: Cache Cargo dependencies
        uses: actions/cache@v3
        with:
          path: |
            ~/.cargo/bin/
            ~/.cargo/registry/index/
            ~/.cargo/registry/cache/
            ~/.cargo/git/db/
            target/
          key: ${{ runner.os }}-bench-${{ hashFiles('**/Cargo.lock') }}

      - name: Install system dependencies
        run: |
          sudo apt-get update
          sudo apt-get install -y netcdf-bin libnetcdf-dev libhdf5-dev

      - name: Run benchmarks
        run: cargo bench --bench extraction_bench --bench filter_bench --bench postprocess_bench --bench combination_bench -- --output-format=bencher | tee output.txt

      - name: Store benchmark result
        uses: benchmark-action/github-action-benchmark@v1
        with:
          name: nc2parquet Benchmarks
          tool: cargo
          output-file-path: output.txt
          github-token: ${{ secrets.GITHUB_TOKEN }}
          auto-push: ${{ github.event_name == 'push' }}
          alert-threshold: "150%"
          comment-on-alert: true
          fail-on-alert: false
          benchmark-data-dir-path: dev/bench
```

2. **Update CONTRIBUTING.md**: Add a "Benchmark regression" row to the CI pipeline table at line 420:
   ```
   | Benchmark regression | `cargo bench -- --output-format=bencher` (warns on >15% regression) |
   ```

### Patterns to Follow

- Follow the same cache key pattern as `ci.yml` but with a `-bench-` suffix to avoid cache collisions (bench builds use different compiler flags than test builds).
- Install the same system dependencies (`netcdf-bin libnetcdf-dev libhdf5-dev`) as the test job.
- Use the `benchmark-action/github-action-benchmark@v1` action, which is the de facto standard for Criterion + GitHub Actions integration.

### Pitfalls to Avoid

- **Do NOT run benchmarks inside the main `ci.yml` workflow**: Benchmarks take 2-5 minutes and would double the CI time. A separate workflow with path filters avoids running benchmarks on docs-only changes.
- **Do NOT set `fail-on-alert: true`**: GitHub Actions runner performance varies by 5-10% between runs. A 15% threshold with failure would produce flaky CI. Use warning comments instead.
- **Do NOT use `cargo bench` without `--bench` flags**: Running bare `cargo bench` would also try to run any `#[bench]` attributes in the library, which does not exist but adds unnecessary compilation time.
- **Do NOT forget `permissions: contents: write`**: The workflow needs write access to push benchmark data to the `gh-pages` branch.
- **Benchmark with `--output-format=bencher`**: This is a Criterion flag (not a cargo flag). It must come AFTER the `--` separator in the `cargo bench` command.
